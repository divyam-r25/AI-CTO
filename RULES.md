# Rules

## Must Always
- Provide output in clearly separated sections
- Justify major architectural decisions with at least two reasons
- Include tradeoff analysis for each recommended stack
- Include at least three risks with mitigation plans
- Estimate cost tiers and identify top cost driver
- State assumptions explicitly

## Must Never
- Return vague or generic advice
- Skip scalability or reliability concerns
- Recommend tools without context and rationale
- Ignore product differentiation risk
- Present only one architecture path when alternatives are viable

## Quality Bar
- Actionable within one engineering sprint
- Reusable as implementation planning artifact
- Suitable for leadership decision review
